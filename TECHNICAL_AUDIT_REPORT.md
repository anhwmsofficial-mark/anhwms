# ANH WMS 기술 감사 보고서 (Technical Audit)

> **감사일**: 2025년 3월 7일  
> **감사 기준**: 실서비스 운영 가능성, 사고 리스크, 확장성  
> **평가 원칙**: 감정적 평가 배제, 실제 코드/구조 기반

---

## 1. 전체 총평

ANH WMS는 Next.js 15 + Supabase 기반의 창고 관리 시스템으로, **기본 아키텍처와 인증/보안 골격은 잘 갖춰져 있다**. 미들웨어 기반 경로 보호, RBAC(역할별 권한), RLS(행 수준 보안), Rate Limit(공유/주문 임포트), Cron 보호, 업로드 검증 등이 구현되어 있다. 다만 **일부 핵심 기능의 미완성(입고 검수 API 불일치, Mock 데이터 폴백), tenant/org 스코핑의 부분적 적용, API 권한 매트릭스 미검증, 운영 관측성 부족** 등이 실서비스 전에 정리되어야 할 과제다. 스타트업 규모에 맞는 MVP 수준으로 보면 **전체 완성도는 72점** 수준이며, 위험 요소를 제거하고 운영 기반을 보강하면 실서비스 가능 수준에 도달할 수 있다.

## 3. 가장 위험한 구조적 문제 TOP 5

### 1) 입고 검수 페이지 Mock 데이터 폴백 (Critical)

- **위치**: `app/(admin)/admin/inbound/[id]/inspect/page.tsx`
- **내용**: `GET /api/inbound/[id]` 라우트가 없어 404 발생 시 **Mock 데이터로 폴백**하여 화면 표시
- **리스크**: 운영 환경에서 API 실패 시 **가짜 데이터로 검수 진행** 가능 → 재고·입고 데이터 오염
- **권장**: MVP 단계에서 즉시 제거. API 실패 시 에러만 표시하고 Mock 폴백 금지

### 2) 입고 검수 API와 실제 스키마 불일치

- **위치**: `app/api/inbound/[id]/inspect/route.ts`
- **내용**: `inbounds`, `inbound_inspections` 테이블 사용. 실제 운영 모델은 `inbound_shipment` / `inbound_shipment_line` 중심
- **리스크**: 테이블/컬럼 불일치로 **404·500 반복**, 비즈니스 로직 오동작
- **권장**: MVP에서 `inbound_shipment` 기반으로 통합하거나, `inbounds` 사용 시 스키마·마이그레이션 상태 명확히 문서화

### 3) 검색어 직접 삽입으로 인한 잠재적 인젝션

- **위치**: `app/actions/admin/customers.ts`, `brands.ts`, `warehouses.ts`, `products.ts`, `locations.ts` 등
- **내용**: `query.or(\`name.ilike.%${search}%...\`)` 형태로 사용자 입력 `search`를 그대로 삽입
- **리스크**: Supabase PostgREST는 파라미터화되어 있으나, `%`, `_` 등 특수문자로 **의도치 않은 패턴 매칭** 가능
- **권장**: MVP에서 최소한 `%`, `_`, `\` 이스케이프. 나중에 검색 전용 sanitize 유틸 도입

### 4) `/api/inbound/*` 경로에 대한 Admin 권한 미검증

- **내용**: 미들웨어는 `/api/admin/*`만 `canAccessAdmin` 검증. `/api/inbound/[id]/inspect`는 로그인만 필요
- **리스크**: staff/operator 등 **비 Admin 역할**도 입고 검수 API 호출 가능. 역할별 제어 의도와 불일치 가능
- **권장**: MVP에서 입고 검수 API를 `/api/admin/inbound/...`로 이동하거나, `requirePermission`으로 역할 검증 강화

### 5) RLS 정책의 tenant/org 스코핑 불완전

- **내용**: baseline 스키마의 `inbound_shipment`, `inbounds` 등에 `USING (true)` SELECT 정책 잔존. 이후 마이그레이션으로 `rls_whitelist_role`, `rls_can_access_admin`, tenant 기반 정책 추가됨
- **리스크**: 마이그레이션 적용 순서·환경 차이로 **멀티테넌트 데이터 유출** 가능
- **권장**: MVP에서 tenant/org 기반 RLS 적용 여부를 전체 테이블 대상으로 점검하고, 미적용 테이블에 대한 정책 보강

## 6. 요약 및 권장 로드맵

### 즉시 (1~2주, MVP)

1. 입고 검수 Mock 폴백 제거 및 `GET /api/inbound/[id]` 구현
2. 입고 검수 API 스키마 정합성 확보 (`inbound_shipment` 또는 `inbounds` 일원화)
3. 검색어 sanitize 적용
4. 입고 검수 API 권한 검증 강화
5. RLS tenant 스코핑 점검
6. Sentry DSN 설정 및 에러 수집 검증
7. E2E 테스트 CI 연동

### 단기 (1~2개월)

- API 권한 매트릭스 검증
- 공개 API Rate Limit 확대
- 헬스체크 심화
- 크론 실패 알림 채널 확대
- `protectedPaths` 중앙 관리

### 중기 (3개월~)

- 업로드 바이러스 스캔
- 비밀번호 정책·감사 로그 보존 정책
- API 버저닝
- 문서화 동기화
