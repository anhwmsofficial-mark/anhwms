# ANH_WMS Supabase 운영 DB 백업 정책

## 1. PITR(Point-in-Time Recovery) 활성화 여부 확인

### 1.1 점검 일시
- 2026-03-01

### 1.2 점검 방법
- Supabase MCP 도구(`execute_sql`, `get_project_url`)를 이용해 운영 프로젝트 상태 조회를 시도함.

### 1.3 점검 결과
- **자동 점검 실패(활성화 여부 미확정)**  
  MCP 응답 기준으로 프로젝트 참조 URL이 비정상(`https://https://...supabase.co.supabase.co`)으로 반환되어 DB 상태 조회가 실패함.
- 현재 기준으로 PITR 활성화 여부는 **대시보드 수동 확인이 필요**함.

### 1.4 즉시 수동 확인 절차(운영 표준)
1. Supabase Dashboard 접속 후 운영 프로젝트 선택
2. `Database > Backups > Point in time` 메뉴 이동
3. 아래 항목 확인 후 운영 기록에 남김
   - PITR 토글 상태(Enabled/Disabled)
   - Recovery Retention(7/14/28일)
   - Earliest Restore Point, Latest Restore Point
4. 점검 결과를 월간 운영 점검 문서에 업데이트

## 2. 백업 방식

운영 DB 백업은 Supabase 관리형 백업 체계를 따른다.

- **기본 백업(일일 백업)**  
  Supabase가 주기적으로 백업을 생성하며, 시점 단위 복구는 제한적이다.
- **PITR 활성화 시(권장)**  
  물리 백업 + WAL 아카이빙 기반으로 특정 시각(초 단위)에 가까운 복구가 가능하다.
- **운영 원칙**
  - 운영(Production) 환경은 PITR 활성화를 기본값으로 유지한다.
  - 비운영 환경은 비용/중요도에 따라 PITR 적용 여부를 분리한다.
  - 백업 정책 변경(보관기간 변경, PITR 비활성화)은 변경관리 승인 후 수행한다.

## 3. 보관 기간(Retention) 정책

### 3.1 목표
- 서비스 중요 데이터를 대상으로 **최소 14일 복구 가능 구간**을 유지하는 것을 기본 목표로 한다.

### 3.2 기준값
- PITR 활성 시: `14일` 권장 (업무/감사 요구사항에 따라 `28일` 상향 가능)
- PITR 비활성 시: 플랜 기본 일일 백업 보관 기간을 따르되, 운영 DB에는 비활성 상태를 장기 유지하지 않는다.

### 3.3 검토 주기
- 월 1회(복구 테스트와 동일 주기) 보관 기간의 적정성 재검토
- 사고/요건 변경 시 즉시 재평가

## 4. 복구 절차(운영 표준)

## 4.1 복구 트리거
- 데이터 손상, 오배포, 대량 삭제, 논리적 데이터 불일치 등 운영 사고 발생 시

## 4.2 복구 의사결정
1. 장애 범위 및 데이터 손실 허용치(RPO) 확인
2. 복구 목표 시간(RTO) 승인
3. 복구 시점(UTC 기준 timestamp) 확정
4. 서비스 공지 및 쓰기 트래픽 통제(필요 시 read-only 전환)

## 4.3 복구 실행
1. Supabase Dashboard에서 `Database > Backups > Point in time` 이동
2. 복구 대상 시점 선택
3. 최종 영향도 검토 후 복구 실행
4. 복구 완료 후 애플리케이션 연결 상태 및 주요 트랜잭션 정상성 확인

## 4.4 복구 후 검증
- 필수 테이블 row count/핵심 지표 대조
- 주문/재고/사용자 인증 등 핵심 시나리오 스모크 테스트
- 에러 로그, 지연 지표, 데이터 정합성 재확인

## 4.5 사후 조치
- 사고 보고서 작성(원인, 영향, 복구 시점, 예방 대책)
- 정책/모니터링/배포 절차 개선 항목 반영

---

## 5. 월 1회 복구 테스트 프로세스

### 5.1 목적
- 백업 유효성 및 실제 복구 가능성을 정기 검증
- 운영팀의 복구 숙련도 유지

### 5.2 일정
- 매월 첫째 주 수요일 10:00 KST (원칙)
- 운영 일정 충돌 시 해당 주 내 재조정

### 5.3 테스트 절차
1. 테스트 기준 시점 선정(예: T-24h)
2. 사전 체크리스트 수행
   - 최근 배포/마이그레이션 확인
   - 복구 대상 데이터 샘플 정의
3. 복구 실행(가능하면 스테이징/복제 환경 우선)
4. 검증 항목 수행
   - 핵심 API 응답
   - 업무 핵심 데이터 정합성
   - 권한/정책(RLS) 동작
5. 결과 기록
   - 성공/실패
   - 소요 시간(RTO 실측)
   - 복구 시점 오차(RPO 실측)
   - 개선 필요 항목

### 5.4 실패 시 대응
- 즉시 원인 분석 후 5영업일 내 재테스트
- 반복 실패 시 보관 기간 상향 또는 복구 절차 재설계 검토

---

## 6. 역할과 책임(RACI)

- **운영 책임자(Owner)**: 백업 정책 승인, 복구 최종 의사결정
- **DBA/백엔드 담당자(Executor)**: 복구 실행, 검증, 결과 기록
- **서비스 오너(Approver)**: 서비스 중단/재개 승인
- **보안/컴플라이언스(Consulted)**: 감사 요구사항 및 보존정책 검토
