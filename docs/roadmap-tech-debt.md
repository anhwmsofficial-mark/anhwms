# 후순위 기술 부채 및 개선 과제

이번 스프린트에서 구현하지 않고, 후속 과제로 정리한 항목이다.

---

## 1. confirm() 기반 파괴 동작 → 공통 확인 다이얼로그

| 항목 | 내용 |
|------|------|
| **필요성** | `confirm()`은 브라우저 기본 UI로, UX 일관성·접근성 부족 |
| **예상 범위** | 삭제/취소/상태 변경 등 파괴 동작이 있는 화면 전반 |
| **회귀 위험** | 중간 (기존 confirm 호출부 교체 시 로직 누락 가능) |
| **추천 우선순위** | P2 |
| **선행조건** | 공통 Dialog 컴포넌트 또는 Radix Dialog 활용 패턴 확립 |

---

## 2. 대용량 테이블 가상 스크롤 / 서버 필터링

| 항목 | 내용 |
|------|------|
| **필요성** | 수천 행 이상 테이블에서 렌더링·스크롤 성능 저하 |
| **예상 범위** | 주문/입고/재고 목록 등 대용량 리스트 화면 |
| **회귀 위험** | 높음 (기존 페이지네이션·정렬 로직과 충돌 가능) |
| **추천 우선순위** | P2 |
| **선행조건** | API 서버 사이드 필터/정렬/커서 페이지네이션 정비 |

---

## 3. 중복 경로 / 중복 파일 정리

| 항목 | 내용 |
|------|------|
| **필요성** | `app/(admin)/...` vs `app\(admin)\...` 등 경로 표기 혼재, 유사 컴포넌트 중복 |
| **예상 범위** | 전체 app 디렉터리, components |
| **회귀 위험** | 중간 (import 경로 변경 시 누락 가능) |
| **추천 우선순위** | P3 |
| **선행조건** | 영향 범위 파악, 점진적 정리 계획 |

---

## 4. 도메인 서비스 레이어 확장

| 항목 | 내용 |
|------|------|
| **필요성** | API route 내 비즈니스 로직 집중, 재사용·테스트 용이성 부족 |
| **예상 범위** | inbound, orders, inventory 등 핵심 도메인 |
| **회귀 위험** | 높음 (대규모 리팩토링) |
| **추천 우선순위** | P2 |
| **선행조건** | 서비스 인터페이스 설계, 기존 API 동작 보존 검증 |

---

## 5. 핵심 도메인 단위 테스트 추가

| 항목 | 내용 |
|------|------|
| **필요성** | 리팩토링·회귀 방지, CI 신뢰도 향상 |
| **예상 범위** | 서비스 레이어, 유틸 함수, 스키마 검증 |
| **회귀 위험** | 낮음 (신규 추가 위주) |
| **추천 우선순위** | P2 |
| **선행조건** | 서비스 레이어 분리 또는 테스트 가능한 모듈 구조 |
